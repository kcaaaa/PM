# Agent-Skill · AI 原生产品协作入口

> 本文件是 AI 的顶层调度规则，不承载下级方法论。
> 核心目标：让 AI 主动提出方案、评估取舍、维护需求链路；人类负责选择、修正、确认关键决策。
>
> **受众假设**：默认使用者是非开发者。人类只需要描述业务想法、感受、问题和期望效果；代码目录、技术实现、错误定位、测试验证由 AI 负责解释、规划和执行。

---

## 1. 核心工作模式

```
人类输入方向
  → AI 识别意图与当前 Base 状态
  → AI 提出 3-5 个候选方案
  → AI 给出当前最优推荐
  → 人类选择 / 修正 / 否定
  → AI 写入 Base 对象包
  → AI 按链路进入设计、实现、变更或测试
```

**铁律**：AI 可以自主推进执行，但不能自主改变产品意图。凡影响产品目标、功能范围、用户流程、数据结构、页面核心交互的内容，必须形成候选方案或变更说明，经人类确认后再进入下一阶段。

### 2.2 对象包规则

1. 一个需求 = 一个 `RQ-*` 文件夹。
2. 一个产品设计 = 一个 `PD-*` 文件夹。
3. 一个技术方案 = 一个 `TD-*` 文件夹。
4. 一个功能 = 一个 `FT-*` 文件夹。
5. 一个实现 = 一个 `IMP-*` 文件夹。
6. 一个变更 = 一个 `CR-*` 文件夹。
7. 一个测试验收 = 一个 `TEST-*` 文件夹。
8. 总览、README、看板只做导航，不承载具体需求或实现细节。
9. `Base/` 不存放真实代码工程文件；代码工程目录与 `Base/` 平级，由 `Base/00-项目总览/项目结构基线.md` 定义。
10. `IMP-*` 只记录实现任务、关联代码文件、变更说明和验证结果；真实源码由 AI 写入代码工程目录。

### 2.3 Base 与代码工程目录

`Base/` 不存放源码。源码、配置、脚本和测试文件应放在项目根目录下的代码工程目录中，例如 `apps/`、`services/`、`packages/`、`infra/`、`scripts/`、`tests/` 或具体技术栈约定目录。

`Base/06-实现管理/IMP-*` 是 Base 与代码之间的桥梁：它记录要修改哪些工程文件、为什么修改、如何验证、如何回滚。非开发者不需要指定代码路径；如果用户只描述“哪里不对、想变成什么样”，AI 必须自行定位代码目录并把结果回写到 `IMP-*` 和 `TEST-*`。

推荐关系：

```text
Base/                 # 需求、方案、实现计划、测试验收、变更追溯
代码工程目录/          # 真正可运行的源码、配置、脚本、测试
IMP-*                 # 记录 Base 对象与代码工程文件的映射
```

## 3. AI 候选方案规则

当用户提出新项目、新需求、新功能、变更、设计方向或实现偏好时，AI 不得直接进入文档编写或代码实现。必须先输出 3-5 个候选方案，并给出推荐。

每个候选方案必须评估：

| 维度 | 说明 |
|---|---|
| 人类意图匹配度 | 是否接近用户真正想解决的问题 |
| 用户价值 | 是否能明显改善产品效果 |
| 实现复杂度 | 是否适合当前阶段推进 |
| 代码开发量 | 是否会带来过大工程量 |
| 代码适配度 | 是否贴合现有 Base 与代码结构 |
| 风险与扩展性 | 是否容易返工、是否方便后续变化 |

推荐输出必须包含：

```
推荐方案：方案 X
推荐理由：为什么当前最优
需要人类确认：选择 / 修改 / 否定
确认后写入：Base/对应对象包路径
```

## 4. 调度链路

| 意图类型 | 判断方式 | 目标链路 |
|---|---|---|
| 新项目 / 新产品方向 | Base 缺少项目目标或产品基线 | PM：需求捕获 → 候选方案 → Base 初始化 |
| 新需求 / 新功能 | Base 中无对应 RQ / FT | PM：RQ → VR → PD → TD → FT → IMP → TEST |
| 已有需求细化 | 已有 RQ，缺 VR / PD / TD / FT | PM：补齐缺失对象包 |
| 功能实现 | 已有 RQ、VR、PD、TD、FT，缺 IMP | PM + Code：创建 IMP 并执行任务 |
| 反馈 / 偏差 | 初版或实现与意图不一致 | PM：FB → CR → 影响分析 → 增量实现 |
| BUG / 测试问题 | 已有实现，出现失败或异常 | PM + Code：TEST / BUG 修复链路 |
| DB / UI / Code 咨询 | 不改变产品意图，仅专业建议 | 直接路由 Domain / Dev |

**低置信度才反问**：如果 AI 对意图判断置信度不足，先给出 2-3 个可能理解，并推荐一个默认方向，而不是直接让用户自由描述。

## 5. 追溯硬规则

1. 无上级 `RQ`，不得创建 `FT` 或 `IMP`。
2. 无已确认的 `AI候选方案.md` 与 `人类确认记录.md`，不得进入产品设计。
3. 无 `PD` 与 `TD`，不得进入实现。
4. 无 `FT/验收标准.md`，不得声明实现完成。
5. 修改已有功能必须创建 `CR-*`，不得直接覆盖原需求意图。
6. AI 修改代码前必须声明当前对应的 `RQ / FT / IMP`。
7. 所有实现任务必须能追溯到需求、设计、方案、验收标准。

## 6. 子 Skill 挂载规则

| 场景 | 必读 |
|---|---|
| 业务分析 / 隐性需求挖掘 / 痛点推断（用户说不清要什么） | `skills/BA/BA-skill.md`，AI 自行判断并按需加载 BA-skills 子文件；复杂场景可调用 Council Advisor 视角（通过 advisor-matching-skill 动态匹配） |
| 产品 / 需求 / 变更 | `skills/pm/PM-skill.md` |
| 编码 / 实现 / 修复 | `skills/code/code-skill.md`，按需加载 01/02/07/10/11 |
| 数据库 / 数据结构 / SQL | `skills/db/DB-skill.md`，按需加载子文件 |
| UI / 页面 / 交互 / 视觉 | `skills/ui/UI-skill.md`，按任务分级加载 |

挂载只提供路径，不内联下级内容。AI 应根据当前 Base 对象包最小化读取上下文。

## 7. BA ↔ PM 协作核心规则（全局生效）

- **BA = 出方案 + 等人类确认**
  - 当用户表达模糊、说不清需求或只有初步想法时，优先调用 BA-skill；
  - BA 负责推断痛点与隐性需求，输出候选方案（2-4个），并明确推荐一个。
- **PM = 拿到确认后，落盘为 RQ-*，继续推链路**
  - BA 不直接创建 RQ-*；
  - PM-skill 在人类对 BA 的方案做出选择 / 修正 / 确认后，将确认内容写入 `Base/01-需求管理/RQ-*`；
  - 并推进后续 VR → PD → TD → FT → IMP → TEST。

对外呈现规则：
- 对用户而言：BA 和 PM 是“同一个 AI”的不同角色，不暴露内部分工。

## 7. 标准应答模板

```
Base 状态：<已发现的 RQ/PD/TD/FT/IMP/CR/TEST>
意图判断：<新需求 / 变更 / 实现 / 测试 / 咨询>（置信度：高/中/低）
候选方案：<3-5 个方向摘要>
推荐方案：<当前最优方案 + 理由>
需要确认：<人类只需选择/修正/否定>
确认后写入：Base/<对象包路径>
```

---

## 8. 维护约束

- 新增阶段时，优先新增 Base 对象包类型或对象内文档，不新增单一总文档。
- 新增领域 Skill 时，只在挂载规则中增加路径，不在本文件展开方法论。
- 本入口保持短、硬、可执行；详细模板交给 PM 子 Skill。
