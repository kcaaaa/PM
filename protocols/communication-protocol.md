# Agent 通信协议 · 轻量版

> 本协议用于约束 Agent 之间传递信息的最小字段。不要使用复杂类型系统模拟运行时，LLM 只需要清晰、可执行的上下文规则。

---

## 1. 通信目标

所有 Agent 交接必须围绕 `Base/` 对象包进行，确保每一步都能追溯到需求源头。

核心链路：

```text
RQ → VR → PD → TD → FT → IMP → TEST → FB → CR
```

---

## 2. 标准交接格式

Agent 交接时只传递以下信息：

```text
任务类型：新需求 / 产品设计 / 技术方案 / 实现 / 测试 / 反馈 / 变更 / BUG
关联 Base：RQ / PD / TD / FT / IMP / CR / TEST
当前状态：已确认 / 待确认 / 缺失 / 冲突
用户意图：一句话描述
候选方案：如影响产品意图，列 3-5 个
推荐方案：当前最优方案与理由
约束条件：不可改变的产品、技术、代码或时间限制
下一步：要创建、读取或更新的 Base 对象包
需要人类确认：是 / 否；如是，说明确认点
```

---

## 3. 调度规则

| 来源 | 目标 | 条件 |
|---|---|---|
| Orchestrator | PM | 涉及需求、设计、方案、功能、变更、测试 |
| Orchestrator | Dev | 已有完整 `RQ / FT / IMP`，且任务是实现或修复 |
| Orchestrator | Domain | 涉及 DB / UI / 技术方案专业判断 |
| PM | Dev | `IMP-*` 已创建并确认 |
| PM | Domain | `TD-*` 需要专业评审或补充 |
| Dev | PM | 发现需求、设计、方案或验收标准不一致 |
| Domain | PM | 专业建议影响产品意图或功能边界 |

---

## 4. 信息最小化

- 只传递当前任务相关 Base 对象包。
- 不全量复制长文档。
- 不把总览文件当作唯一事实来源。
- 具体需求、方案、功能、实现、变更、测试必须进入独立对象包。

---

## 5. 人类确认点

以下情况必须请求人类确认：

1. 选择候选需求方案。
2. 改变产品目标或功能范围。
3. 改变用户核心流程。
4. 改变数据结构或核心接口。
5. 执行破坏性变更、迁移或大规模重构。
6. 接受或拒绝 `CR-*` 变更。

---

## 6. 禁止事项

- 禁止无 `RQ` 直接进入功能或实现。
- 禁止无 `CR` 覆盖历史需求。
- 禁止用 Agent 内部判断替代人类产品确认。
- 禁止用复杂接口、枚举、类定义替代可执行规则。
