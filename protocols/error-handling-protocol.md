# Agent 错误处理协议 · 轻量版

> 错误处理目标：先保护 Base 链路与用户意图，再决定自动修复、降级处理或请求人类确认。

---

## 1. 错误分类

| 类型 | 判断 | 处理 |
|---|---|---|
| 上下文缺失 | 找不到对应 `RQ / FT / IMP` | 停止实现，退回 PM 补 Base |
| 产品意图不清 | 用户描述有多种解释 | 输出 3-5 个候选方案并推荐 |
| 需求冲突 | 新需求与已确认 RQ / CR 冲突 | 列冲突点，给合并/取舍方案 |
| 实现偏差 | 代码实现与 Base 不一致 | 创建 FB 或 CR，做影响分析 |
| 技术阻塞 | 架构、依赖、性能、安全存在风险 | 给 2-4 个技术方案并推荐 |
| 测试失败 | 验证不通过 | 记录 TEST 问题清单，定位后修复 |
| 破坏性风险 | 删除、迁移、重构、数据变更 | 必须请求人类确认 |

---

## 2. 处理优先级

```text
保护用户意图
→ 保护 Base 链路
→ 保护现有代码和数据
→ 尝试最小自动修复
→ 给候选方案
→ 请求人类确认
```

---

## 3. 自动处理规则

可以自动处理：

- 明确的格式、命名、引用路径问题。
- 不改变产品意图的小范围代码错误。
- 测试中暴露的纯实现缺陷。
- Base 对象包内的状态同步和索引更新。

必须先确认：

- 改变产品目标。
- 改变功能范围。
- 改变核心流程或数据结构。
- 删除、迁移、覆盖已有文件或数据。
- 将一个需求拆分、合并或废弃。

---

## 4. 标准错误响应

```text
问题类型：<上下文缺失 / 产品意图不清 / 实现偏差 / 技术阻塞 / 测试失败>
影响对象：<RQ / PD / TD / FT / IMP / CR / TEST>
当前风险：<一句话说明>
可选处理：<2-4 个方案>
推荐处理：<当前最优方案 + 理由>
是否需要人类确认：是 / 否
下一步：<更新 Base / 修复代码 / 创建 CR / 补充测试>
```

---

## 5. 禁止事项

- 禁止为了继续执行而跳过缺失的 `RQ / FT / IMP`。
- 禁止把需求偏差当作 BUG 静默修复。
- 禁止在没有影响分析时执行变更。
- 禁止在测试失败时声明完成。
- 禁止用复杂错误码系统替代清晰说明。
