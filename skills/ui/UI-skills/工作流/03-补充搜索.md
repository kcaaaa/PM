# 工作流 Step 3 · 补充搜索

> **职责**：在已有设计系统候选方案后，按 Base 链路需要补充图表、UX、图标、性能、Web 规范等细节。

---

## 1. 触发矩阵

| 触发条件 | 打开的知识库 | 应写入的 Base 位置 |
|---|---|---|
| 数据可视化、Dashboard、报表 | 《06-图表选型.md》 | `PD/页面流程.md`、`TD/方案说明.md`、`FT/验收标准.md` |
| 动画、滚动触发、无障碍体验 | 《07-UX规范.md》 | `PD/交互规则.md`、`FT/验收标准.md` |
| 图标体系或具体图标 | 《08-图标库.md》 | `PD/交互规则.md`、`IMP/前端实现.md` |
| React / Next / SSR / 性能问题 | 《09-React性能.md》 | `TD/方案说明.md`、`IMP/前端实现.md` |
| ARIA、焦点、键盘、语义化 | 《10-Web界面规范.md》 | `FT/验收标准.md`、`TEST/测试用例.md` |
| 备选风格 / 配色 / 字体 / 页面结构 | 02/03/04/05 | `PD` 或 `CR` |

---

## 2. 选择原则

1. 已确认的 `RQ / FT` 优先于风格偏好。
2. 与 Anti-Patterns 冲突的候选一律剔除。
3. 影响性能或无障碍的候选，必须说明风险。
4. 若补充建议会改变页面核心流程，必须创建或更新 `CR-*`。
5. 若补充建议只影响实现细节，应写入 `TD-*` 或 `IMP-*`。

## 4. 完成后

- 只涉及设计 → 回写 `PD-*`。
- 涉及实现方式 → 转《工作流/04-技术栈应用.md》并写入 `TD-*` 或 `IMP-*`。
- 涉及功能边界或体验目标变化 → 创建或更新 `CR-*`。
