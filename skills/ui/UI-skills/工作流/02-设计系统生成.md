# 工作流 Step 2 · 设计系统生成

> **职责**：基于 UI 信号与 Base 链路，生成 2-4 个设计系统候选方案，并推荐最适合当前产品阶段的方案。

---

## 1. 输入

- Step 1 输出的 UI 四元组。
- 当前关联的 `RQ / PD / TD / FT / IMP / CR`。
- 已确认的产品目标、功能边界、验收标准。

---

## 2. 总体算法

```
A. 产品定位  → 查《01-产品-风格映射》定位 UI_Category
B. 推理调权  → 取 Style_Priority / Anti_Patterns / Decision_Rules
C. 多域取值  → 查 02/03/04/05 得到 Style / Color / Font / Pattern
D. 候选生成  → 生成 2-4 个 UI 方向
E. 推荐决策  → 结合复杂度、开发量、适配度、风险选最优
F. Base 写入 → 输出应写入 PD / TD / FT / IMP / TEST 的内容
```

---

## 3. 产品定位

打开《知识库/01-产品-风格映射.md》，按以下优先级找 `UI_Category`：

1. 精确匹配 `product_type`。
2. 子串匹配。
3. 关键词打分。
4. 兜底到 `SaaS (General)`。

记下：`UI_Category / Recommended_Pattern / Style_Priority / Color_Mood / Typography_Mood / Key_Effects / Decision_Rules / Anti_Patterns / Severity`。

---

## 4. 候选方案规则

至少输出 2 个，最多 4 个 UI 方向：

| 方案 | 适合场景 | 实现复杂度 | 代码开发量 | 代码适配度 | 风险 | 推荐度 |
|---|---|---|---|---|---|---|
| A | MVP / 标准 / 增强 / 高端 | 高/中/低 | 高/中/低 | 高/中/低 | 描述 | 推荐/可选/不建议 |

推荐方案必须优先满足：

1. 不违反 Base 中已确认的人类意图。
2. 不改变 `RQ / FT` 功能边界。
3. 与现有技术基线和代码结构适配。
4. 能被 `FT/验收标准.md` 与 `TEST-*` 验证。

---

## 5. 决策合并优先级

```
Base 已确认产品意图
  ↑ 最高优先级
Anti_Patterns
must_have 规则
FT 验收标准
Style_Priority 首位风格
Color_Mood / Typography_Mood
style_keywords
Performance / Accessibility
```

---

## 6. 输出到 Base

根据阶段输出到对应文件：

| 阶段 | 写入位置 |
|---|---|
| 产品设计 | `Base/03-产品设计/PD-*/页面流程.md`、`交互规则.md`、`设计验收.md` |
| 技术方案 | `Base/04-方案设计/TD-*/方案说明.md` 或 UI 技术方案章节 |
| 功能定义 | `Base/05-功能管理/FT-*/功能边界.md`、`验收标准.md` |
| 实现建议 | `Base/06-实现管理/IMP-*/前端实现.md` |
| 测试验收 | `Base/09-测试验收/TEST-*/测试用例.md` |
| 变更 | `Base/08-变更管理/CR-*/影响分析.md`、`AI处理建议.md` |

## 8. 禁止事项

- 禁止无 Base 关联直接输出落地级代码。
- 禁止 UI 方案改变已确认产品意图。
- 禁止只给一个方案且不说明取舍。
- 禁止跳过验收标准和测试映射。
