# UI-design-skill · Base 约束下的 UI/UX 领域能力

> **定位**：本 Skill 是 UI/UX 领域专家，服务于 `Base` 对象包链路，不是独立产品流程。
>
> **核心原则**：UI 可以提供风格、交互、设计系统与实现建议，但不得脱离 `RQ / PD / TD / FT / IMP / CR` 改变产品意图。
>
> **数据规模**：67 种 UI 风格、96 套配色、57 组字体、100 条产品推理规则、99 条 UX 规范、13 套技术栈指南。

---

## 1. 触发条件

用户请求涉及以下语义时加载本 Skill：

- 页面、组件、落地页、App、Dashboard、信息架构、交互流程。
- UI 审查、视觉优化、体验优化、无障碍、响应式、设计系统。
- 风格、配色、字体、图标、图表、页面结构、CTA 策略。
- 技术栈相关 UI 落地：Tailwind、React、Next、Vue、Svelte、shadcn、SwiftUI、Flutter 等。

---

## 2. Base 前置判断

处理前先判断当前 UI 请求属于哪类：

| 场景 | Base 要求 | 是否可直接输出代码 |
|---|---|---|
| 纯咨询：配色 / 字体 / 风格建议 | 可直接回答，但建议说明可写入 PD/TD | 否 |
| 产品设计：页面流程 / 交互规则 / 信息架构 | 必须绑定 `RQ-*`，输出到 `PD-*` | 否 |
| 功能设计：组件状态 / 异常交互 / 验收标准 | 必须绑定 `FT-*`，更新功能边界与验收标准 | 否 |
| 技术落地：组件结构 / 样式实现建议 | 必须绑定 `TD-*` 或 `IMP-*` | 仅 `IMP-*` 已确认时可以 |
| 反馈或体验偏差 | 必须绑定 `FB-*` 或 `CR-*` | 否，先做影响分析 |

**硬规则**：无 `RQ / PD / FT / IMP` 时，不得直接输出落地代码；应先建议创建或补齐对应 Base 对象包。

## 3. 调度流程（按需加载）

```
[Step 0] 判断 Base 关联对象：RQ / PD / TD / FT / IMP / CR / TEST
[Step 1] 读《UI-skills/工作流/01-需求分析.md》→ 提取 UI 信号 + Base 链路
[Step 2] 读《UI-skills/工作流/02-设计系统生成.md》→ 生成设计系统候选方案
[Step 3] 读《UI-skills/工作流/03-补充搜索.md》→ 按需补充图表、UX、图标、性能、Web 规范
[Step 4] 读《UI-skills/工作流/04-技术栈应用.md》→ 仅在 TD/IMP 阶段生成技术落地建议
[Step 5] 读《UI-skills/检查清单/通用UI规范.md》+《交付前检查.md》→ 对齐 FT 验收标准与 TEST
[Step 6] 输出：设计建议 / PD 更新 / TD 更新 / IMP 实现建议 / TEST 检查项
```

> 轻量咨询可直达知识库，但如果建议会影响产品流程、页面结构或实现方式，必须回到 Base 链路。

## 5. 核心不变约束

1. **绝不用 Emoji 当 icon**，使用 SVG（Heroicons / Lucide / Simple Icons）。
2. 所有可点击元素必须有 `cursor-pointer`；悬停态不得造成 layout shift。
3. 浅色模式正文对比度 ≥ 4.5:1。
4. 过渡 150–300ms；超过 500ms 或瞬时切换都不合格。
5. Logo / 品牌图标必须来自 Simple Icons 官方 SVG，不许凭记忆画。
6. 政府 / 医疗 / 金融 / 企业场景禁用 AI 流行色“紫粉渐变”。
7. 响应式断点必测 375 / 768 / 1024 / 1440；移动端不允许横向滚动。
8. `prefers-reduced-motion` 必须被尊重。
9. UI 方案必须能追溯到 Base 对象包；无法追溯时，只能作为咨询建议，不能进入实现。

- 新增工作流规则 → 必须说明写入哪个 Base 对象包。
