# 任务管理 Task Management

> 当前项目适配：TodoWrite 用于会话内执行进度管理，`Base/06-实现管理/IMP-*` 与 `Base/09-测试验收/TEST-*` 用于项目级持久追溯。Todo 完成不等于 Base 实现完成；代码任务结束后仍需把变更、验证和遗留问题回写到对应对象包。

## 1. 何时使用 Todo 列表

**主动使用**于以下场景：

1. **复杂多步任务**——需要 3+ 个明确步骤或动作。
2. **非琐碎/复杂任务**——需要仔细规划或多处操作。
3. **用户明确要求** Todo 列表。
4. **用户一次性给出多个任务**（编号列表、逗号分隔列表）。
5. **收到新指令后**——立即把用户需求捕获为 Todo。
6. **开始一个任务时**——在**开始干活之前**标记为 `in_progress`。理想情况同时只有一个 `in_progress`。
7. **完成一个任务后**——立即标记 `completed`，并添加实现中发现的后续任务。

## 2. 何时**不**使用

**跳过**以下场景：

1. 只有一个直截了当的任务。
2. 琐碎任务，追踪它没有组织化收益。
3. 少于 3 步就能完成的琐碎任务。
4. 纯对话性/信息性请求（解释 `git status` 作用等）。

**判断规则**：只有一个琐碎任务时，**直接做**比记录它更好。

## 3. 正例与反例

**正例 - 适合 Todo**：
- 新增深色模式切换 + 跑测试 + 构建（多阶段功能 + 显式后续步骤）。
- 在 8 个文件中重命名 `getCwd` → `getCurrentWorkingDirectory`（多文件系统性改动）。
- 实现：注册、商品目录、购物车、结算（多个复杂功能并列）。
- 优化 React 应用性能（需要先分析再列出多处优化点）。

**反例 - 不适合**：
- "如何在 Python 中打印 Hello World？"（单步、信息性）。
- "git status 作用是什么？"（纯解释）。
- "给 calculateTotal 加一行注释"（单点修改）。
- "帮我跑 npm install"（单命令即时结果）。

## 4. 任务状态

三种状态：

| 状态 | 含义 |
|---|---|
| `pending` | 尚未开始 |
| `in_progress` | 正在进行（**最多同时一个**） |
| `completed` | 成功完成 |

**任务两种形式必须同时提供**：
- `content`：**祈使式**描述要做什么（"Run tests"、"Build the project"）。
- `activeForm`：**现在进行时**，执行中显示（"Running tests"、"Building the project"）。

## 5. 状态流转硬规则

1. **实时更新**：干活时就更新状态，不要批量。
2. **完成立即标记**：任务一结束立即标 `completed`，不要攒起来一起标。
3. **恰好一个 in_progress**：不能 0 个，不能 2 个。
4. **开始新任务前完成当前任务**。
5. **不再相关的任务**：直接从列表移除。

## 6. 完成的条件

**只有完全完成才能标 `completed`**。遇到任何以下情况，**保持 `in_progress`**：
- 测试失败
- 实现不完整
- 遇到未解决的错误
- 找不到必要文件或依赖

**被阻塞时**：新建一个任务描述"要解决什么才能继续"。**绝不**把未完成/有 bug 的工作标为 `completed`。

## 7. 任务拆分原则

- **具体、可执行**条目。
- 复杂任务**拆成更小、可管理**的步骤。
- 清晰、描述性的任务名。
- 始终提供**祈使式 + 现在进行时**两种形式。

## 8. 与其他工具的协作

- **初次创建 Todo 时**，除了告知用户你在做什么，**倾向于把第一个 Todo 设为 `in_progress`，并在同一轮工具调用批次中直接开始做**。
- **与其他工具批量**：Todo 写入应与其他独立工具调用并行，以降低延迟。
- 在 git commit 工作流等受限场景中，**禁止使用 TodoWrite 或 AgentTool** (参见 Git 工作流技能)。
- 用户修正或追加指令时，用 `merge=true` 增量更新，**不要**重新创建整个列表。

## 9. 原则总结

> 不确定时，**就用**。主动任务管理展现专注度，并确保完整完成所有要求。

**但不要过犹不及**：
- 不要因为"要显得周到"而给 2 步的活建 5 条 Todo。
- 不要把"测试这个改动"作为自动补上的 Todo，除非用户要求。
