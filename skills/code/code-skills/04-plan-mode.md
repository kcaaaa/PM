# 计划模式 Plan Mode

## 1. 核心动机

**在非平凡实现任务开始前获得用户签字同意**——避免白做功、确保方向一致。计划模式让你先探索代码库、设计实现方案、呈现给用户审批。

当前项目采用 `Base/` 对象包作为计划与执行边界：如果任务会改变产品意图、功能范围、数据结构、接口契约或页面核心交互，应先进入 PM 链路生成或更新 `RQ / PD / TD / FT / CR / IMP`，再执行代码工作。

**进入计划模式需要用户同意**，模式本身不会改文件。

## 2. 何时使用 (偏向使用)

实现任务默认倾向于进入计划模式，只要满足以下**任一**条件：

### 2.1 新功能实现
有意义的新功能。例子：
- "加一个登出按钮"——放在哪？点击后做什么？
- "加表单校验"——什么规则？错误消息？

### 2.2 多种可行方案
任务可用几种不同方式解决。例子：
- "给 API 加缓存"——Redis / 内存 / 文件？
- "改善性能"——多种优化策略。

### 2.3 代码修改
影响现有行为或结构的改动。例子：
- "更新登录流程"——具体改什么？
- "重构这个组件"——目标架构？

### 2.4 架构决策
需要在模式/技术间选择。例子：
- "加实时更新"——WebSockets / SSE / 轮询？
- "实现状态管理"——Redux / Context / 自研？

### 2.5 多文件改动
任务预计要改 2-3 个以上文件。例子：
- "重构认证系统"
- "新增 API 端点+测试"

### 2.6 需求不清
需要先探索才能理解完整范围。例子：
- "让 app 更快"——需要先 profile 找瓶颈。
- "修 checkout 的 bug"——需要调查根因。

### 2.7 用户偏好要紧
实现可以合理走多种方向。**如果你本打算用 AskUserQuestion 问方案选择，那就直接进入计划模式**——计划模式让你可以先探索再带上下文呈现选项。

## 3. 何时**不**使用

只在**简单任务**跳过计划模式：

- **单行或少数几行修复**（打错字、明显 bug、小调整）。
- **需求清晰的单个函数添加**。
- **用户给了非常具体、详细指令**时。
- **纯研究/探索**任务（用 Explore agent，而非计划模式）。

## 4. 偏保守（ant 内部版变体）

对于更成熟/自主的使用者：
- 只在**真正方案歧义明显**且用户反馈会防止显著返工时进入计划模式。
- **用户的"我们来做 X 吧"、"让我们搞 X"这种口吻 → 直接开始**，不要先进计划模式。
- 存疑时，**偏向"开始干 + 特定问题用 AskUserQuestion 澄清"** 而不是进入完整计划阶段。

**决策启发**：如果写个方案的价值 > 花费的沟通开销，就进计划；如果用户已经表达"让我们开始"的动作意图，就尊重它。

## 5. 计划模式中的流程

进入计划模式后：

1. **彻底探索代码库**（使用 Glob、Grep、Read、或 Explore agent）。
2. **理解现有模式与架构**。
3. **设计实现方案**。
4. **将方案呈现给用户审批**。
5. 需要时用 `AskUserQuestion` **澄清具体选择**（不要问"方案 OK 吗？"）。
6. **用 `ExitPlanMode`** 在准备开始实现时退出。

## 6. 计划模式中的沟通纪律

- **不要用 AskUserQuestion 问"方案准备好了吗？"或"要继续吗？"**——用 `ExitPlanMode` 让用户审批。
- **不要在问题中提及"the plan"**（"对方案有反馈吗？"、"方案看起来好吗？"）——用户在你调用 `ExitPlanMode` 之前看不到方案。
- 计划模式中的 `AskUserQuestion` **只用于**在**最终确定方案前**澄清需求或在方法间做选择。

## 7. 典型计划结构

呈现给用户的方案应包含：

1. **需求确认**：你理解的任务、成功标准。
2. **实现方案**：步骤、涉及文件、关键决策。
3. **关键取舍**：为何选这个方案而非其他。
4. **关键文件**：3-5 个最重要的文件路径。

## 8. Plan Agent (委派规划)

对于需要更深入规划的情况，可用 Agent 工具委派给 **Plan 子代理** (`subagent_type="Plan"`)：
- Plan 是只读架构师角色。
- 返回分步实现策略、关键文件清单、架构取舍。
- 不能创建/修改文件。

## 9. 默认倾向

**不确定是否该用 → 偏向使用**。上下文与对齐的前置成本低，返工成本高。
