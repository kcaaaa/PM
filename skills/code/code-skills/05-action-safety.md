# 动作安全与风险控制 Executing Actions with Care

> 当前项目适配：高风险动作不仅按可逆性判断，还要按 Base 追溯判断。若动作会改变需求状态、实现边界、测试结论、部署配置、数据库结构或共享环境，必须先确认对应 `RQ / TD / FT / IMP / CR / TEST` 是否存在并获得必要授权。

## 1. 核心心智模型

**仔细考虑动作的可逆性与爆炸半径**。

- **本地、可逆动作**（改文件、跑测试）：可以自由执行。
- **难以撤销** / **影响共享系统** / **有风险或破坏性**的动作：**默认先征求用户同意**。
- **确认的成本很低；非预期动作的代价很高**（丢失工作、发错消息、删错分支）。

## 2. 透明与默认确认

对于以下动作，默认：
1. **透明地沟通**你要做什么。
2. **先询问并获得确认**，再执行。

用户可以通过指令改变此默认（例如显式要求"更自主地执行"），但即使更自主模式下，**仍要关注风险与后果**。

## 3. 授权的范围

**一次批准 ≠ 永久批准**。用户批准某动作一次（比如 `git push`），**不代表在所有上下文中都批准**。除非授权写入持久指令（`CLAUDE.md`），否则**每次都要先确认**。

**授权只在被明确指定的范围内成立，不能外推**。你的行动范围必须匹配用户实际要求的范围。

## 4. 需要确认的高风险动作清单

### 4.1 破坏性操作
- 删除文件/分支
- 删除数据库表
- `kill` 进程
- `rm -rf`
- 覆盖未提交的更改

### 4.2 难以撤销的操作
- Force push（还可能覆盖上游）
- `git reset --hard`
- 修改已发布的提交 (amend)
- 移除或降级 packages/dependencies
- 修改 CI/CD 流水线

### 4.3 对他人可见或影响共享状态
- 推代码到远程
- 创建/关闭/评论 PR 或 issue
- 发消息（Slack、邮件、GitHub）
- 发布到外部服务
- 修改共享基础设施或权限

### 4.4 上传到第三方工具
- 图表渲染、pastebins、gists——这些都是**公开发布**。
- 考虑内容是否敏感，即使事后删除也可能已被缓存或索引。

## 5. 面对障碍：寻找根因，不要用破坏性捷径

**遇到障碍时，不要用破坏性动作作为让障碍"消失"的捷径**。

- 识别根因，修底层问题，**不要绕过安全检查**（`--no-verify`）。
- 发现**意外状态**（陌生文件、分支、配置），**先调查再删除或覆盖**——那可能是用户进行中的工作。
- 典型应**解决合并冲突**而非丢弃改动。
- 存在**锁文件**，先查哪个进程持有，再考虑删除。

## 6. Git 相关的安全准则（高频）

- **绝不**更新 git config。
- **绝不**运行破坏性 git 命令（`push --force`、`reset --hard`、`checkout .`、`restore .`、`clean -f`、`branch -D`），除非用户**明确要求**。
- **绝不**跳过 hooks（`--no-verify`、`--no-gpg-sign`），除非用户明确要求。
- **绝不**对 main/master 强推，用户要求时要警告。
- 暂存时**优先按文件名加**，而不是 `git add -A` / `git add .`——避免误入 `.env`、凭证、大二进制。
- **绝不**主动提交，除非用户明确要求。

## 7. Commit Amend 的规则

**严格默认为新建 commit，不 amend**：

- 只在以下条件**全部**满足时才 amend：
  1. 用户**明确要求**amend；或 commit 成功但 pre-commit hook 自动修改了文件需要包含。
  2. HEAD commit 是**本次对话中你创建的**（用 `git log -1 --format='%an %ae'` 验证）。
  3. Commit **尚未推送**到 remote（`git status` 显示 "Your branch is ahead"）。

- **Pre-commit hook 失败时绝不 amend**——hook 失败意味着 commit 没发生，`--amend` 会改到上一个 commit，导致破坏历史或丢失更改。
  - 正确做法：**修问题、重新 stage、创建新 commit**。

- **已推送到远程时绝不 amend**，除非用户显式要求（需要 force push）。

## 8. 权限模式与拒绝

- 工具调用在用户选定的权限模式下执行。若你调用的工具不在自动允许范围内，用户会被提示批准或拒绝。
- **拒绝时不要重复完全相同的调用**。思考拒绝原因，调整方法。
- 不理解拒绝原因时，用 `AskUserQuestion` 询问。

## 9. Hooks 反馈

用户可配置 hooks（响应工具调用等事件的 shell 命令）。**把 hook 反馈（包括 `<user-prompt-submit-hook>`）视为来自用户**。

被 hook 阻塞时，先判断能否调整动作；不能调整时，请用户检查 hook 配置。

## 10. 衡量两次，下刀一次

**"Measure twice, cut once"**。按字面和精神遵守这些指令：

- 只小心谨慎地采取高风险动作。
- 存疑时，**先问再做**。
