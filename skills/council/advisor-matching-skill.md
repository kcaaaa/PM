# advisor-matching-skill · 动态匹配专家视角

> 目标：让 AI 基于“需要什么能力”去选人，而不是基于“谁有名”去套用风格。

## 一、核心原则

- **按能力维度选 Advisor**，不按人名硬编码。
- **组合优于单人**：复杂问题优先使用多 Advisor 组合视角。
- **动态读取注册表**：每次匹配前读取 `advisors.md`，基于其“领域”和“适合议题”字段进行匹配。
- **对外不暴露人名**：BA/PM 对用户的输出只体现能力维度（如“从风险与不确定性角度…”），内部再映射到具体 Advisor。

## 二、输入要素

调用本 Skill 时，AI 应明确以下信息（可从上下文推断）：

- `task_type`：任务类型
  - 例如：痛点推断 / 候选方案评估 / MVP验证 / 风险识别 / 商业模式设计 / 组织管理分析 / AI技术选型等。
- `context`：关键上下文
  - 行业、阶段（早期/成熟）、风险等级、利益相关方数量与冲突程度。

## 三、能力维度定义

根据常见 BA/PM 任务，抽象出以下能力维度（内部使用）：

| 能力维度 | 说明 |
|----------|------|
| 产品体验与定位 | 用户体验取舍、品牌定位、功能克制与聚焦 |
| 创业策略与验证 | MVP设计、Build-Measure-Learn、客户开发、早期增长 |
| 风险与不确定性 | 黑天鹅应对、反脆弱、概率思维、逆向思考 |
| AI/技术架构 | LLM应用架构、AI基础设施、技术选型判断 |
| 商业模式与竞争 | 错位竞争、垄断战略、网络效应、生态布局 |
| 组织与管理 | 企业文化、领导力、危机管理、长期主义经营 |

## 四、匹配逻辑（内部）

1. **读取注册表**：
   - 读取 `advisors.md`，获取所有 Advisor 的“领域”和“适合议题”。
2. **按能力维度筛选候选池**：
   - 根据 `task_type` + `context`，选择需要的能力维度（1-3个为主）。
   - 从注册表中挑选与这些维度匹配的 Advisor。
3. **组合策略**：

| 场景 | 推荐方式 |
|------|----------|
| 单一明确问题（如“MVP怎么设计”） | 选择 1-2 个最匹配该能力维度的 Advisor |
| 多利益相关方冲突 / 高风险决策 | 选择 3-4 个不同视角的 Advisor，形成圆桌讨论 |
| AI/技术强相关需求 | 优先加入 AI/技术架构维度匹配的 Advisor |

4. **输出格式（内部使用）**：

返回结构如下（不展示给用户）：

```markdown
- 任务类型：{task_type}
- 所需能力维度：[维度1, 维度2]
- 推荐组合：
  - {Advisor A}（侧重：XX维度）
  - {Advisor B}（侧重：YY维度）
- 使用方式：单 Advisor / 圆桌讨论
```

## 五、调用关系

- **BA-skills**：在需要专家视角时，先调用本 Skill 获取推荐组合，再按结果加载对应 persona.md。
- **PM-skill**：在复杂需求/变更评估中也可调用本 Skill。
- **对外呈现规则**：
  - 对用户只说“从XX角度分析”（如“从风险与不确定性角度…”），不暴露具体人名。
