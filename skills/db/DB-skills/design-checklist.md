# 数据库设计检查清单

> 适用范围：本清单由 `skills/db/DB-skill.md` 控制调用，用于数据库需求分析、技术方案、实现任务、变更评估、测试验收等场景。
>
> 核心原则：保留通用数据库设计能力，同时必须服从当前工作流中的 Base 链路与可追溯要求。任何可执行 SQL、迁移脚本、生产优化动作，必须绑定对应 `RQ / PD / TD / FT / IMP / CR / TEST` 对象包。

---

## 0. 调用前置检查（必须先做）

- [ ] 已识别当前场景类型：概念咨询 / 新功能设计 / 结构变更 / 性能优化 / 数据迁移 / 测试验证。
- [ ] 已确认是否属于项目落地内容；若只是概念咨询，仅输出示例，不包装为可执行方案。
- [ ] 涉及新业务建模时，已关联 `RQ-*`、`PD-*`、`TD-*`。
- [ ] 涉及字段、索引、约束设计时，已关联 `RQ-*`、`FT-*`、`TD-*`。
- [ ] 涉及建表脚本、迁移脚本、回滚方案、生产执行顺序时，已关联 `IMP-*`。
- [ ] 涉及已确认方案变化时，已创建或更新 `CR-*`。
- [ ] 涉及验证、压测、回归时，已创建或更新 `TEST-*`。
- [ ] 已明确本次输出写入位置：如 `数据库设计.md`、`数据库实现.md`、`影响分析.md`、`测试用例.md`。

---

## 1. 技术栈适配检查

> DB-skills 不应默认绑定某个业务项目或单一数据库产品。若上游 `TD-*`、技术栈文档或用户明确指定数据库类型，则按指定技术栈输出；否则应先澄清或给出多数据库差异说明。

- [ ] 已确认目标数据库类型：MySQL / PostgreSQL / Oracle / SQL Server / SQLite / NewSQL / NoSQL / 混合架构。
- [ ] 未将某一数据库的专属语法当作通用 SQL 输出。
- [ ] 若输出 MySQL 示例，已标注 MySQL 适用范围，如 `AUTO_INCREMENT`、`ENGINE=InnoDB`、`ON UPDATE CURRENT_TIMESTAMP`。
- [ ] 若输出 PostgreSQL 示例，已标注 PostgreSQL 适用范围，如 `BIGSERIAL`、`GENERATED AS IDENTITY`、`JSONB`、`GIN`、`TIMESTAMPTZ`。
- [ ] 若输出 Oracle / SQL Server 示例，已标注序列、身份列、分页、时间函数等差异。
- [ ] 若涉及 Redis、Elasticsearch、MongoDB、Kafka、对象存储等组件，已明确它们与关系型数据库的职责边界。
- [ ] 若技术栈未确定，输出应停留在设计建议、选型对比或待确认问题，不直接给生产脚本。

---

## 2. 需求分析阶段

- [ ] 已识别业务模块、核心实体与数据边界。
- [ ] 已明确实体生命周期：创建、修改、审核、发布、归档、删除、恢复等。
- [ ] 已明确高频操作：新增、编辑、查询、统计、导出、批量处理、审计写入等。
- [ ] 已确认性能要求：并发量、响应时间、吞吐量、数据增长量、可用性指标。
- [ ] 已确认安全等级、数据分类分级、隐私与审计要求。
- [ ] 已识别敏感信息、凭证、日志、审计记录等合规数据。
- [ ] 已明确数据保留周期、归档策略、备份恢复要求。
- [ ] 已评估数据量级、增长趋势、冷热数据分布和容量规划。

---

## 3. 概念设计阶段

- [ ] 完成实体识别和分类。
- [ ] 确定实体属性、主从关系与聚合边界。
- [ ] 绘制 ER 图或等价实体关系说明。
- [ ] 识别实体间关系：1:1、1:N、M:N、树形结构、流程状态流转。
- [ ] 覆盖主要业务场景、异常场景、管理场景和审计场景。
- [ ] 与 `RQ / PD / FT` 中的业务描述保持一致。
- [ ] 对不确定的数据边界明确列为待确认问题，而不是直接假设。

---

## 4. 逻辑设计阶段

- [ ] ER 模型已转换为关系模型。
- [ ] 每张表均有明确业务含义，不存在无归属表。
- [ ] 主键、唯一键、业务编码、外键逻辑已明确。
- [ ] 多对多关系已通过关联表表达。
- [ ] 字典、状态、枚举、流程状态已统一口径。
- [ ] 历史记录、审计记录、操作日志、状态流转记录的保存方式已明确。
- [ ] 如存在冗余字段，已说明来源、同步策略、一致性风险和修复手段。
- [ ] 至少满足第三范式；因性能或查询便利而反规范化时，已写明原因。

---

## 5. 物理设计阶段

### 5.1 表结构设计

- [ ] 表名符合命名规范，且能体现模块边界。
- [ ] 字段名使用小写字母、数字、下划线。
- [ ] 主键策略统一，且与项目 ID 生成方案一致。
- [ ] 字段类型符合目标数据库能力，不混用其他数据库专属类型。
- [ ] 字段长度、精度、默认值合理。
- [ ] 所有表和字段均有注释或说明。
- [ ] 包含必要审计字段；审计字段命名与项目规范一致。
- [ ] 软删除字段与业务状态字段未混淆。
- [ ] 状态字段已定义取值范围及含义。
- [ ] 敏感字段已明确加密、脱敏、哈希或不可存储策略。
- [ ] 文件、图片、视频等大对象优先存对象存储或文件服务引用，不直接存入业务表大字段，除非设计明确要求。

### 5.2 索引设计

- [ ] 主键自动创建唯一索引。
- [ ] 高频 WHERE 条件字段有索引。
- [ ] JOIN 关联字段有索引。
- [ ] ORDER BY / GROUP BY / 分页游标字段有索引。
- [ ] 复合索引顺序符合目标数据库优化器与实际查询模式。
- [ ] 索引类型选择合理：B-tree、Hash、全文索引、GIN、GiST、BRIN、位图索引、向量索引等按数据库能力选择。
- [ ] 单表索引数量受控，写多读少表避免过度索引。
- [ ] 已对核心 SQL 设计执行计划验证方式，如 `EXPLAIN` / `EXPLAIN ANALYZE`。
- [ ] 已识别可能导致索引失效或低效的写法。

### 5.3 约束设计

- [ ] 主键约束已设置。
- [ ] 唯一约束已覆盖业务唯一字段。
- [ ] 非空约束与默认值合理。
- [ ] 检查约束适用于状态、范围、金额、时间等字段。
- [ ] 外键约束已按项目策略选择：数据库强约束或应用层逻辑约束。
- [ ] 删除、归档、级联策略清晰。

### 5.4 存储与容量设计

- [ ] 大表已评估分区、分表、归档或冷热分离方案。
- [ ] 已规划索引空间、表空间、备份空间与增长余量。
- [ ] 已评估读写分离、连接池、缓存、搜索引擎分担压力的必要性。
- [ ] 已定义归档与清理策略，避免无限增长影响性能。

---

## 6. 安全与合规设计

- [ ] 已识别敏感数据和隐私数据。
- [ ] 密码、密钥、Token 等凭证不明文存储。
- [ ] 敏感字段已明确加密、哈希、脱敏或访问控制策略。
- [ ] 访问控制遵循最小权限原则。
- [ ] 重要操作支持审计追踪。
- [ ] 日志记录满足查询、追溯、导出和保留周期要求。
- [ ] 已考虑 SQL 注入防护、参数校验、越权访问防护。
- [ ] 备份数据、导出数据、测试数据已考虑脱敏与权限隔离。

---

## 7. 性能设计

- [ ] 已对关键读写场景进行容量与连接数评估。
- [ ] 已对核心接口或核心查询进行索引、缓存、异步或预计算设计。
- [ ] 已对报表、日志、导出类重查询设计异步化、离线统计或只读副本方案。
- [ ] 已识别热点数据并设计缓存策略。
- [ ] 已识别搜索场景并设计搜索引擎、全文索引或专用查询方案。
- [ ] 已设计慢查询监控、容量告警和性能基线。

---

## 8. 实现与迁移检查

- [ ] 建表脚本、变更脚本、回滚脚本分别提供。
- [ ] 脚本按执行顺序编号，避免不可重复执行。
- [ ] DDL 与 DML 分离。
- [ ] 生产数据迁移前具备备份方案。
- [ ] 大表变更具备灰度、分批、在线变更或停机窗口说明。
- [ ] 迁移后提供行数校验、抽样校验、业务校验和性能校验。
- [ ] 所有实现内容写入 `IMP-*/数据库实现.md`。

---

## 9. 测试验收检查

- [ ] 提供数据库对象创建验证用例。
- [ ] 提供字段约束、唯一约束、状态流转验证用例。
- [ ] 提供核心查询性能测试用例。
- [ ] 提供缓存一致性、搜索同步、消息幂等验证用例（如适用）。
- [ ] 提供备份恢复、权限控制、安全审计验证用例。
- [ ] 提供回归测试清单，并写入 `TEST-*`。
- [ ] 测试结果不得编造；实测数据缺失时标注“待实测”。

---

## 10. 输出自检

- [ ] 输出没有把示例 SQL 伪装成生产可执行脚本。
- [ ] 输出明确说明适用数据库类型。
- [ ] 输出未夹带无关业务项目的信息。
- [ ] 输出保留通用模板能力，同时遵守 Base 链路约束。
- [ ] 输出已标注引用或依赖的 `RQ / PD / TD / FT / IMP / CR / TEST`。
- [ ] 输出包含风险、回滚、验证和验收口径。
