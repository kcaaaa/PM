# PROJECT-CONTEXT · Base 上下文协议

> 在任何项目相关工作前，先理解 `Base/` 与代码结构。Base 是 AI 的项目认知数据库。

---

## 1. 必须先检查的内容

1. `Base/00-项目总览/`：项目目标、产品状态、阶段进度、决策记录。
2. `Base/01-需求管理/`：当前任务相关的 `RQ-*` 需求包。
3. `Base/03-产品设计/`：与需求对应的 `PD-*` 产品设计包。
4. `Base/04-方案设计/`：与功能对应的 `TD-*` 技术方案包。
5. `Base/05-功能管理/`：与实现对应的 `FT-*` 功能包。
6. `Base/06-实现管理/`：与代码任务对应的 `IMP-*` 实现包。
7. `Base/08-变更管理/`：是否存在影响当前任务的 `CR-*`。
8. 代码目录：只扫描与当前对象包相关的模块，不做无目的全量读取。

---

## 2. 最小上下文原则

- 新需求：优先读取 `00-项目总览/` 与相近 `RQ-*`。
- 产品设计：读取对应 `RQ-*` 与 `VR-*`。
- 技术方案：读取对应 `RQ-*`、`PD-*` 与相关代码结构。
- 功能实现：读取对应 `RQ-*`、`PD-*`、`TD-*`、`FT-*`、`IMP-*`。
- 变更：读取原始链路对象与对应 `CR-*`。
- BUG / 测试：读取 `IMP-*`、`TEST-*` 与相关代码。

---

## 3. 追溯检查

执行前必须能回答：

1. 当前任务对应哪个 `RQ`？
2. 是否已有候选方案与人类确认记录？
3. 是否已有产品设计、技术方案、功能定义？
4. 如果要改代码，对应哪个 `IMP`？
5. 如果是变更，是否已有 `CR`？

如果无法回答，不得直接进入实现，应先补齐 Base 对象包。

---

## 4. 输出要求

在开始执行前简要说明：

```
已读取 Base：<对象包列表>
当前链路：RQ → VR → PD → TD → FT → IMP → TEST → FB → CR
缺失项：<如有>
下一步：<补齐对象包 / 进入实现 / 做变更分析>
```

---

## 5. 禁止事项

- 禁止用单一总文档替代对象包。
- 禁止无 `RQ` 直接做功能或实现。
- 禁止无 `CR` 直接覆盖已有需求。
- 禁止为了简单任务全量加载 Base。
