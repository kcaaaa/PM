# 功能需求澄清 · 从需求到功能对象包

> 新功能不得直接进入实现。必须先确认需求来源、候选方案、功能边界与验收标准。

---

## 1. 输入

- 用户提出的新功能、体验优化或能力补充。
- `Base/01-需求管理/RQ-*` 中相关需求。
- 已有相近 `Base/05-功能管理/FT-*`，用于判断是否重复或属于变更。

---

## 2. 工作流程

1. 扫描相关 `RQ-*`，判断新功能是否已有上级需求。
2. 若无上级 `RQ`，先回到项目/需求澄清，不得创建 `FT`。
3. 输出 3-5 个功能方案，覆盖 MVP、标准、增强、长期扩展等方向。
4. 综合人类意图、用户价值、实现复杂度、代码开发量、代码适配度、风险扩展性推荐最优方案。
5. 人类确认后创建或更新 `FT-*` 功能包。
6. 若该功能是对已有功能的调整，转入 `CR-*` 变更包，不覆盖原 `FT`。

---

## 3. 输出对象包

```
Base/05-功能管理/FT-xxx-功能名称/
├── 功能说明.md
├── 功能边界.md
├── 异常场景.md
├── 验收标准.md
└── 关联关系.md
```

详细产物模板见：`./FT-TEMPLATE.md`

- CR-xxx

---

## 6. 禁止事项

- 禁止无 `RQ` 创建 `FT`。
- 禁止把多个功能写进一个总功能清单。
- 禁止没有验收标准就进入实现。
- 禁止把功能变更直接覆盖原功能说明。
