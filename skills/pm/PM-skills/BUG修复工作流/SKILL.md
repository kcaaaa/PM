# BUG 修复工作流 · Base 可追溯修复

> BUG 修复不是孤立动作，必须绑定受影响的 `FT / IMP / TEST`，必要时创建 `CR`。

---

## 1. 前置检查

修复前必须定位：

- 发生在哪个功能：`FT-*`
- 影响哪个实现：`IMP-*`
- 是否已有测试：`TEST-*`
- 是否涉及需求或设计变化：如涉及，创建 `CR-*`

若无法定位功能来源，先创建反馈记录 `Base/07-反馈管理/FB-xxx-问题描述.md`，不得直接改代码。

---

## 2. 工作流程

1. 收集问题描述、复现步骤、期望结果、实际结果、环境、日志或截图。
2. 创建或更新 `TEST-* / 问题清单.md`。
3. 复现问题；未复现不得修改代码。
4. 定位根因，判断是否只是 BUG，还是需求/设计偏差。
5. 若是需求/设计偏差，创建 `CR-*` 并等待确认。
6. 若是纯 BUG，按最小改动修复。
7. 更新或新增自动化测试。
8. 输出手动验证步骤。
9. 更新 `IMP/实现完成记录.md` 与 `TEST/测试记录.md`。

---

## 3. 输出对象

优先更新：

```
Base/09-测试验收/TEST-xxx-功能名称测试/
├── 测试用例.md
├── 测试记录.md
├── 问题清单.md
└── 验收结论.md
```

如涉及变更，创建：

```
Base/08-变更管理/CR-xxx-问题修复或需求调整/
```

详细产物模板见：`./BUG-TEMPLATE.md`

---

## 4. 修复报告要求

修复完成后在 `TEST/测试记录.md` 中记录：

- 问题现象。
- 复现步骤。
- 根因分析。
- 修改文件。
- 自动化测试结果。
- 手动验证步骤。
- 是否影响需求、设计、方案或验收标准。

---

## 5. 禁止事项

- 未复现不得修改代码。
- 无 `FT / IMP / TEST` 不得直接修复。
- 需求偏差不得伪装成 BUG 静默修复。
- 修复后不得只口头说明，必须更新 Base 测试或变更对象包。
