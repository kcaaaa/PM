# 项目需求澄清 · AI 候选方案驱动

> 人类不需要一次性说清完整需求。AI 必须先展开可能性，再让人类选择、修正或否定。

---

## 1. 输入

- 用户的项目方向、问题描述、业务想法或初步目标。
- 已存在的 `Base/00-项目总览/` 与相近 `Base/01-需求管理/RQ-*`。

---

## 2. 工作流程

1. 扫描 `Base/00-项目总览/`，确认项目目标、阶段进度、历史决策。
2. 判断是否已有相近 `RQ-*`，避免重复创建需求。
3. 基于用户意图提出 3-5 个候选项目 / 需求方向。
4. 综合人类意图、用户价值、实现复杂度、代码开发量、代码适配度、风险扩展性，推荐当前最优方向。
5. 等待人类选择、修正或否定。
6. 人类确认后创建或更新 `Base/01-需求管理/RQ-xxx-需求名称/`。

---

## 3. 输出对象包

确认后必须创建需求包：

```
Base/01-需求管理/RQ-xxx-需求名称/
├── 需求说明.md
├── AI候选方案.md
├── 人类确认记录.md
└── 需求状态.md
```

不得把多个需求写进同一个集合文档。

详细产物模板见：`./RQ-TEMPLATE.md`

---

## 4. AI候选方案.md 模板

```
# RQ-xxx 需求候选方案

## 原始人类意图
<用户原始描述>

## 候选方案

### 方案 A：<名称>
- 适合场景：
- 人类意图匹配度：高/中/低
- 用户价值：高/中/低
- 实现复杂度：高/中/低
- 代码开发量：高/中/低
- 代码适配度：高/中/低
- 风险与扩展性：
- 结论：推荐 / 可选 / 不建议

## 当前最优推荐
- 推荐方案：
- 推荐理由：
- 需要人类确认：选择 / 修改 / 否定
```

---

## 5. 需求说明.md 模板

```
# RQ-xxx 需求名称

## 1. 需求来源

## 2. 目标用户

## 3. 核心问题

## 4. 用户场景

## 5. 需求边界

## 6. 非目标范围

## 7. 验收标准

## 8. 上下级关系
- 上级：项目目标 / 决策记录
- 下级：VR / PD / TD / FT / IMP / TEST
```

---

## 6. 禁止事项

- 禁止直接问“你具体想做什么”后等待用户补全全部需求。
- 禁止未给候选方案就创建需求。
- 禁止多个独立需求混写在同一个文档。
- 禁止无 `人类确认记录.md` 进入产品设计。
