# PM 调度交互示例

> 仅在用户问“怎么用 / 给我看个例子”或 AI 意图不明确时加载。

---

## 示例 1：新项目启动

```text
用户：我想做个记账 App，但还没想清楚。

PM-skill：
Base 状态：未发现项目总览与 RQ 对象包
意图判断：新项目 / 新需求（置信度：高）

候选方案：
1. 极简记账 MVP：手动记账 + 分类统计，开发量低
2. 预算管理型：预算目标 + 超支提醒，用户价值高
3. 家庭共享型：多人账本 + 权限协作，复杂度中高
4. 自动化增强型：票据识别 + AI 分类，开发量高

推荐方案：方案 1
推荐理由：最符合“先做清楚”的意图，开发量低，后续可平滑扩展到方案 2/4。

需要确认：选择 / 修改 / 否定
确认后写入：Base/01-需求管理/RQ-001-记账MVP/
```

## 示例 2：给现有项目加功能

```text
用户：帮我加一个评论功能。

PM-skill：
Base 状态：已发现项目目标；未发现评论相关 RQ / FT
意图判断：新功能（置信度：高）

候选方案：
1. 基础评论：发布、删除、列表展示
2. 互动评论：点赞、回复、排序
3. 审核评论：敏感词、举报、后台审核
4. 社区评论：楼中楼、通知、个人主页聚合

推荐方案：方案 1
推荐理由：最小闭环清晰，开发量低，适合先形成初版；方案 2/3 可作为后续 CR。

需要确认：是否以方案 1 创建 RQ 与 FT？
确认后写入：
- Base/01-需求管理/RQ-xxx-评论功能/
- Base/05-功能管理/FT-xxx-评论功能/
```

## 示例 3：已完成功能要改

```text
用户：评论功能加个点赞按钮。

PM-skill：
Base 状态：已发现 RQ-评论功能、FT-评论功能、IMP-评论功能实现
意图判断：已有功能变更（置信度：高）

候选策略：
1. 作为小变更立即实现：新增点赞字段、接口和按钮
2. 拆成互动能力新需求：点赞、回复、通知一起规划
3. 延后到互动增强版本：先不影响当前 MVP

推荐策略：方案 1
推荐理由：开发量可控，与现有评论功能适配度高，不改变核心产品方向。

需要确认：是否创建 CR？
确认后写入：Base/08-变更管理/CR-xxx-评论点赞/
```

## 示例 4：报 BUG

```text
用户：登录页点击登录按钮没反应。

PM-skill：
Base 状态：需定位对应 FT / IMP / TEST
意图判断：BUG / 测试问题（置信度：高）

下一步：
1. 定位登录功能对应 FT 与 IMP
2. 创建或更新 TEST/问题清单.md
3. 先复现，再修复
4. 如发现是需求或交互偏差，创建 CR

需要补充：发生环境、复现步骤、期望结果与实际结果。
```
