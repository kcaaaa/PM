# PM 实战准则 · 10 条

> 仅在用户问“最佳实践 / 经验 / 窗口管理 / AI 协作技巧”时由 PM-skill 引导加载。

1. **Base 是上下文锚点** — 规划、设计、实现可以分窗口进行，但每个窗口必须先读取相关 `Base` 对象包，不依赖聊天历史。

2. **人类做选择，AI 给方案** — 不要求人类一次说清需求；AI 先给 3-5 个候选方案并推荐最优，人类选择、修正或否定。

3. **一个对象一个包** — 一个需求、一个功能、一个实现、一次变更、一次测试都必须独立成包，避免大文档混杂。

4. **先追溯再实现** — 编码前必须能说清当前对应的 `RQ / FT / IMP`；说不清就退回 PM 链路补齐。

5. **变更不覆盖历史** — 需求变化必须创建 `CR-*`，保留原始需求和决策记录，避免 AI 把历史意图改没。

6. **AI 生成的 Base 扫两分钟** — 重点看候选方案、推荐理由、功能边界、验收标准是否符合你的真实想法。

7. **用反馈驱动修正** — 初版不符合预期时，不要直接让 AI 重写；先创建 `FB-*` 或 `CR-*`，再做影响分析。

8. **优先小闭环** — 新需求优先选择 MVP 或标准方案，增强能力作为后续 CR，减少一次性开发过重。

9. **一次只推进一条链路** — 多个需求应拆成多个 `RQ-*`，多个变更应拆成多个 `CR-*`。

10. **AI 搞不定按此顺序处理** — 检查 Base 链路 → 补候选方案 → 拆小对象包 → 降低实现范围 → 再进入代码。
