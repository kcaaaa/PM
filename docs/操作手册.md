# 全流程产品 Agent · 操作手册

> **版本**：2.0.0  
> **适用对象**：非开发者、产品经理、业务负责人、创业者、AI 编程用户  
> **目标**：让 AI 从人类模糊意图出发，生成候选方案、维护 Base 对象包，并按需求链路推进到实现和测试。
>
> **默认假设**：使用者不需要懂代码、不需要知道文件应该改哪里，也不需要描述技术实现方式。人类只需说明"想要什么、哪里不对、希望变成什么样"，AI 负责把自然语言转化为 `RQ → VR → PD → TD → FT → IMP → TEST → FB → CR` 对象包、代码修改和验证记录。

---

## 0. 系统简介（快速了解）

本系统是 **AI 原生产品协作工具链**，核心组成：

- **Orchestrator（协调器）**：识别你的真实意图，扫描 Base 状态，生成候选方案并推荐最优解。
- **PM-Agent（产品流程）**：维护 RQ→VR→PD→TD→FT→IMP→TEST→FB→CR 链路，确保需求可追溯。
- **Dev-Agent（开发执行）**：基于 IMP-* 实现包执行代码任务，保证每次修改都可追溯到对应需求。
- **Domain-Agent（领域专家）**：数据库/SQL/UI/性能等垂直领域的专业支持。

你只需要用自然语言描述想法或问题；AI 会自动选择合适 Agent、生成方案、推进链路。

---

## 1. 核心理念

全流程产品 Agent 不再要求人类一次性写清完整需求，而是采用：

```text
人类输入方向
→ AI 生成 3-5 个候选方案
→ AI 推荐当前最优方案
→ 人类选择 / 修正 / 否定
→ AI 写入 Base 对象包
→ 按 RQ → VR → PD → TD → FT → IMP → TEST → FB → CR 链路推进
```

人类负责判断方向，AI 负责展开方案、评估取舍、维护链路和执行。

---

## 2. Base 是项目认知数据库

`Base/` 是 AI 的项目基准库。它不是单一文档集合，而是由独立对象包组成。

```text
Base/
├── 00-项目总览/              # 项目目标、状态看板、阶段进度、决策记录
├── 01-需求管理/RQ-xxx-名称/   # 每个需求一个文件夹
├── 02-验证管理/VR-*.md        # 需求验证记录
├── 03-产品设计/PD-*/          # 页面流程、用户路径、交互规则
├── 04-方案设计/TD-*/          # 接口设计、数据结构、安全策略
├── 05-功能管理/FT-*/          # 功能边界、异常场景、验收标准
├── 06-实现管理/IMP-*/         # 实现任务清单、代码文件索引、变更说明
├── 07-反馈管理/FB-*.md        # 用户反馈与偏差记录
├── 08-变更管理/CR-*/          # 影响分析、处理策略、执行记录
└── 09-测试验收/TEST-*/        # 测试用例、问题清单、验收结论
```

---

## 3. 对象包规则

| 对象 | 含义 | 规则 |
|---|---|---|
| RQ | 需求 | 每个需求一个文件夹，包含 AI 候选方案与人类确认记录 |
| VR | 验证 | 每个需求验证一份记录，确保需求可执行、无歧义 |
| PD | 产品设计 | 每个功能或需求一个设计包，明确页面流程与交互规则 |
| TD | 技术方案 | 每个功能一个技术方案包，定义接口、数据结构与安全策略 |
| FT | 功能 | 每个功能一个功能包，包含边界、异常场景与验收标准 |
| IMP | 实现 | 每个功能实现一个实现包，记录任务清单与代码文件索引 |
| FB | 反馈 | 每次反馈一份记录，用于偏差分析与后续变更 |
| CR | 变更 | 每次需求或功能变化一个变更包，不直接覆盖原需求 |
| TEST | 测试 | 每个功能一个测试验收包，包含用例、问题清单与结论 |

总览、README、看板只做导航，不承载具体需求、功能或实现细节。

---

## 4. Base 与代码工程目录

`Base/` 不存放真实源码。它记录需求、设计、方案、实现任务、测试结果和变更原因；真正可运行的代码、配置、脚本、测试文件放在与 `Base/` 平级的代码工程目录中。

推荐关系：

```text
项目根目录/
├── Base/                 # AI 项目认知库：需求、方案、任务、验收、变更
├── apps/                 # 前端、后台、移动端等应用代码
├── services/             # 后端服务、接口服务、AI 服务等代码
├── packages/             # 公共组件、工具、配置
├── infra/                # Docker、K8s、CI/CD、部署配置
├── scripts/              # 自动化脚本
└── tests/                # 测试代码
```

非开发者不需要判断代码应该放在哪个目录。AI 在进入实现阶段时必须：

1. 读取 `Base/00-项目总览/项目结构基线.md`。
2. 根据 `IMP-*` 判断要改哪些工程文件。
3. 自行扫描代码工程目录并定位文件。
4. 修改完成后，把关联文件、验证结果和遗留问题写回 `IMP-*` 与 `TEST-*`。

## 5. 日常使用方式

### 5.1 新需求

你可以只说方向：

```text
我想做一个面向中小企业的客户跟进工具
```

AI 应先输出：

```text
Base 状态：当前无相近 RQ
意图判断：新需求 / 新产品方向
候选方案：3-5 个
推荐方案：当前最优方案
需要确认：选择 / 修改 / 否定
确认后写入：Base/01-需求管理/RQ-xxx-客户跟进工具/
```

### 5.2 新功能

```text
帮我加一个客户标签功能
```

AI 不应直接编码，而应检查是否已有上级 `RQ`，然后生成候选功能方案，并在确认后创建 `FT-*`。

### 5.3 功能实现

```text
开始实现客户标签功能
```

AI 必须确认：

```text
RQ → VR → PD → TD → FT → IMP → TEST 是否完整
```

缺少任一对象包时，应先补齐，不得直接写代码。

### 5.4 需求变化

```text
客户标签功能我想改成支持多级标签
```

AI 不应直接覆盖原需求，而应创建：

```text
Base/08-变更管理/CR-xxx-客户标签支持多级/
```

并输出影响分析、处理策略和推荐方案。

### 5.5 代码问题或报错

非开发者可以直接描述现象：

```text
点击保存后没有反应。
```

```text
这个页面打开很慢。
```

```text
我不知道哪里出错了，你帮我检查并修复。
```

AI 必须把这类描述转化为可执行的排查链路：

```text
现象理解：<用户描述的问题>
可能原因：<2-4 个方向>
推荐排查顺序：<从低风险到高风险>
需要确认：是否允许 AI 扫描代码和运行验证
确认后动作：定位 RQ / FT / IMP / TEST，必要时创建 CR
```

用户不需要提供文件路径、函数名、接口名或数据库表名；这些由 AI 在代码工程目录中自行定位。

## 6. 人类如何把控项目

人类不需要管理 AI 的每一步，只需要把控以下闸口：

1. **候选方案**是否符合真实意图。
2. **推荐方案**是否可接受。
3. 是否允许写入 Base。
4. 是否允许从需求进入设计。
5. 是否允许从方案进入实现。
6. 变更是否值得执行。
7. 测试结果是否可验收。

---

## 7. 标准链路

```text
RQ 需求
→ VR 验证
→ PD 产品设计
→ TD 技术方案
→ FT 功能定义
→ IMP 实现任务
→ TEST 测试验收
→ FB 反馈
→ CR 变更
→ IMP 增量实现
→ TEST 回归测试
```

任何代码实现都必须能追溯到 `RQ / FT / IMP`。

---

## 8. AI 必须遵守的硬规则

1. 新需求必须先给 3-5 个候选方案。
2. AI 必须推荐当前最优方案，并说明理由。
3. 无 `RQ` 不得创建 `FT` 或 `IMP`。
4. 无人类确认不得改变产品意图。
5. 变更必须创建 `CR`，不得直接覆盖原文档。
6. 代码实现必须绑定 `RQ / FT / IMP`。
7. 总文档只做索引，具体内容必须进入独立对象包。

## 10. 推荐对话方式

推荐说法：

```text
我想做一个 XX，但还没想清楚，你先给我几个方案并推荐一个。
```

```text
选方案 B，但保留方案 C 的数据统计能力。
```

```text
这个实现不符合我的想法，请创建变更并分析影响。
```

不推荐说法：

```text
直接帮我写代码。
```

除非对应 `RQ / PD / TD / FT / IMP` 已经完整。
